# 方向

本文用于约束后续 agent 的实现方向。不要把它扩写成产品愿景，也不要在第一版引入本文未允许的复杂度。

## 1. 最终形态

做一个自托管 PWA 网站。

使用方式：

```text
iPhone Safari 打开网站
添加到主屏幕
像 App 一样使用
```

不做：

- 原生 iOS App
- Apple Developer Program
- App Store
- TestFlight
- iOS 签名和审核

## 2. 运行路径

固定路线：

```text
Mac 写代码
Mac npm run dev 调试
Mac npm run build / npm run start 测生产模式
Mac Docker Compose 测部署
N100 Docker Compose 长期运行
Tailscale Serve 暴露 HTTPS
iPhone 通过 Tailscale 访问 PWA
```

不要一开始就在 N100 上边写边调。

## 3. 技术栈

第一版使用：

- Next.js
- TypeScript
- Tailwind CSS
- SQLite
- Drizzle ORM
- Zod
- Recharts
- PWA
- Docker Compose
- Tailscale

第一版不要使用：

- Postgres
- Redis
- Kubernetes
- 原生 iOS
- 多用户系统
- 复杂云同步
- 本地大模型
- 复杂 AI Agent

## 4. 产品分层

系统按下面顺序构建：

```text
数据收集层
↓
结构化层
↓
分析层
↓
决策层
```

不要跳过结构化层直接做复杂分析。

## 5. 第一版功能边界

第一版只做：

- 快速记录
- 记录保存
- 时间线展示
- 自动标签
- 每日总结
- 每周趋势
- 决策建议
- 导出 / 导入
- PWA 添加到主屏幕

第一版不做：

- 登录系统
- 多用户
- 云同步
- 日历集成
- 推送通知
- AI 聊天
- 复杂图表
- 本地大模型

第一版目标只有一个：验证“记录 -> 分析 -> 决策”闭环是否有价值。

## 6. 页面

第一版页面固定为：

- `/`：Dashboard
- `/record`：快速记录
- `/timeline`：记录时间线
- `/insights`：分析结果
- `/decisions`：行动建议
- `/settings`：设置、导出、导入

## 7. 数据流

前端不能直接访问数据库。

固定数据流：

```text
React 页面
↓
Next.js Route Handler
↓
lib/db
↓
SQLite
```

用户记录后的主流程：

```text
POST /api/records
↓
写入 SQLite
↓
结构化处理
↓
生成 insights
↓
生成 decisions
↓
Dashboard 展示
```

## 8. API

第一版 API 只需要：

- `POST /api/records`
- `GET /api/records`
- `GET /api/insights`
- `GET /api/decisions`
- `PATCH /api/decisions/:id`
- `GET /api/export`
- `POST /api/import`

不要提前设计复杂 API。

## 9. 数据库

SQLite 放在服务器端，不放在 iPhone IndexedDB。

开发环境：

```text
./data/app.db
```

生产环境：

```text
/opt/personal-os/data/app.db
```

第一版表的方向：

- `records`
- `structured_records`
- `insights`
- `decisions`
- `settings`

可以先从 `records`、`insights`、`decisions` 开始。

禁止：

- 把数据库文件放进 Docker 容器内部
- 写死只能在某台机器运行的绝对路径
- 提交 `data/`、`backups/`、`.env`

## 10. 部署

N100 目标目录：

```text
/opt/personal-os/
  app/
  data/
    app.db
  backups/
  docker-compose.yml
  .env
```

Docker Compose 要求：

```yaml
ports:
  - "127.0.0.1:3000:3000"
volumes:
  - ./data:/data
  - ./backups:/backups
```

端口只绑定本机，由 Tailscale Serve 提供 HTTPS。

不要把服务直接暴露到公网。

## 11. 医学与分析边界

这个系统不是诊断工具，不输出医学结论。

允许输出：

- 趋势
- 记录摘要
- 疑似关联
- 可验证的行动建议

禁止输出：

- 确诊判断
- 治疗建议
- 单日因果结论
- “某食物必然导致银屑病加重”这类确定性表述

分析层优先使用简单事件窗口：

```text
某暴露出现后
观察 D+0 胃肠反应
观察 D+1 到 D+3 皮肤变化
同时参考压力、睡眠、排便
```

鼻部症状只能作为并行变量记录，不默认归因于肠漏。

## 12. 阶段顺序

按顺序做：

1. Mac 本地开发：Next.js、SQLite、记录 API、记录页面
2. 核心闭环：结构化、insights、decisions、Dashboard
3. 生产模式测试：`npm run build`、`npm run start`
4. Docker 化：Dockerfile、docker-compose.yml、SQLite volume
5. N100 部署：Ubuntu Server、Docker Compose、Tailscale Serve
6. PWA 完善：manifest、icon、移动端优化、导出 / 导入 / 备份

不得跳过生产模式测试和 Docker 测试直接迁移 N100。

## 13. 总约束

Mac 负责开发，N100 负责长期运行，iPhone 负责使用。

产品只服务一个目标：

```text
低成本持续记录
↓
结构化个人数据
↓
发现稳定模式
↓
给出可执行建议
```
